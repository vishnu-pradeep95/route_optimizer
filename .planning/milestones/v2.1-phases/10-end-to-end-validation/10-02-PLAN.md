**FULL REWRITE -- do NOT copy-paste from old LICENSING.md.** Use the actual codebase (`license_manager.py`, `enforcement.py`, `get_machine_id.py`, `build-dist.sh`) as the source of truth.

Preserve the good structural patterns from the old doc (overview, developer guide, customer guide, lifecycle, troubleshooting) but rewrite ALL content. Use Claude's discretion for audience structure (developer-only vs dev+customer split).

**Required sections (minimum):**

1. **Overview** -- Hardware-bound license keys, what happens without a valid key (503 on all endpoints except /health). ASCII diagram of the flow (keep the existing style but update content).

2. **For Developers: Generating a License Key**
   - Step 1: Get fingerprint -- customer runs `get_machine_id.py`. Output is SHA256 hash of `/etc/machine-id` + CPU model (NOT hostname+MAC+container_id).
   - Step 2: Generate key -- `generate_license.py` with all flags. Include `--renew` flag if it exists.
   - Step 3: Send key to customer.

3. **For Customers: Activating a License**
   - Option A: LICENSE_KEY env var (recommended)
   - Option B: license.key file
   - Verifying activation: `curl /health` and check `license.status` field in body + `X-License-Expires-In` header

4. **License Renewal** (NEW section -- was missing)
   - **Simple file drop renewal:** Generate renewal.key with `generate_license.py --renew`, customer places file at `renewal.key` in project root, restart API. No new fingerprint exchange needed.
   - **Full re-key renewal:** If fingerprint changed, follow full generation process.
   - Document both workflows clearly.

5. **Monitoring and Diagnostics** (NEW section)
   - `X-License-Expires-In` header on all responses (e.g., `X-License-Expires-In: 45d`)
   - `X-License-Warning` header during grace period
   - `X-License-Status: invalid` header when expired
   - `/health` endpoint body includes `license` section with status, expires_at, days_remaining, fingerprint_match

6. **Grace Period** -- 7-day window after expiry. Status transitions: VALID -> GRACE -> INVALID.

7. **Dev vs Production**
   - Dev: Full source (.py), `ENVIRONMENT=development` must be explicitly set (production is the default now per Phase 6), license not enforced in dev mode
   - Production: Compiled to `.so` via Cython (NOT `.pyc`), license required, enforcement via compiled module
   - Build pipeline: `build-dist.sh` strips dev code -> hashes files -> compiles to .so -> validates imports -> packages tarball

8. **Security Architecture** (NEW section)
   - **Integrity checking:** SHA256 manifest of protected files embedded in compiled .so, verified at startup and every 500 requests
   - **Periodic re-validation:** License and integrity re-checked every 500 requests (offline, no network calls)
   - **Enforcement module:** Single `enforce(app)` entry point in compiled .so, main.py has no inline enforcement
   - **HMAC signing:** PBKDF2 key derivation with 200k iterations, seed stored as bytes.fromhex() (not human-readable)

9. **Troubleshooting** -- Update all troubleshooting entries:
   - 503 "License expired or invalid" -- check key, check expiry
   - 503 fingerprint mismatch -- run get_machine_id.py, note that fingerprint uses machine-id + CPU model (stable across container recreation)
   - "File integrity check failed" -- protected files modified, reinstall from distribution tarball
   - Invalid key format -- check for whitespace, verify starts with LPG-

10. **License Lifecycle** -- Update the ASCII diagram. Add renewal.key flow. Show monitoring headers.

**CRITICAL -- content that MUST NOT appear:**
- hostname, MAC, container_id in fingerprint references
- .pyc or "bytecode" compilation references
- "ENVIRONMENT=development (default)" -- it is NOT the default anymore
- Any suggestion that HMAC seed is human-readable or stored as a string
- "compile to bytecode" -- it's Cython .so now
