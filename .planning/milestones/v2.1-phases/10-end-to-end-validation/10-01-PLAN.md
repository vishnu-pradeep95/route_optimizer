1. **e2e/security-pipeline.spec.ts** -- Create a new Playwright spec file with 4 serial test groups. Follow the patterns from `e2e/license.spec.ts` (execSync for docker compose, polling for container readiness). Use `test.describe.serial()` for each scenario group.

   **General patterns:**
   - Base URLs: security test on port 8002, fingerprint test on port 8003
   - All containers started in production mode (ENVIRONMENT=production)
   - Generate license keys in beforeAll using `execSync('python scripts/generate_license.py ...')` -- capture stdout to get the key string
   - Use `docker exec --user root` when modifying files inside containers (appuser cannot write to application files)
   - Container readiness: poll /health endpoint with exponential backoff (reuse pattern from license.spec.ts)
   - Set generous timeouts: `test.setTimeout(120_000)` for container lifecycle tests

   **Scenario 1: Fingerprint Mismatch Rejection**
   - beforeAll: Generate a license key with fingerprint `"0" * 64` (will not match any machine). Use Python inline: `execSync('python -c "from core.licensing.license_manager import encode_license_key; ..."')` to create a key bound to fake fingerprint. Set as `FINGERPRINT_TEST_LICENSE_KEY` env var. Start `api-fingerprint-test` container.
   - Test: GET /api/routes returns 503. GET /health returns 200 with `X-License-Status: invalid` header.
   - afterAll: Stop and remove `api-fingerprint-test` container.

   **Scenario 2: Periodic Re-Validation Triggering**
   - beforeAll: Generate a valid license key for this machine using `execSync('python scripts/generate_license.py --customer "security-e2e" --this-machine --months 1 --verify')`. Parse the key from stdout. Set as `SECURITY_TEST_LICENSE_KEY` env var. Start `api-security-test` container (REVALIDATION_INTERVAL=10). Wait for container to be ready and return 200 on /health.
   - Test 1: Verify API is working -- GET /api/routes returns 200 (or appropriate success status).
   - Test 2: Send 10+ requests to trigger re-validation cycle. Verify API still works (re-validation with valid license should pass silently). Check that no errors occurred.
   - afterAll: Stop and remove `api-security-test` container.

   **Scenario 3: Integrity Tamper Detection**
   - beforeAll: Start `api-security-test` with valid license (same as Scenario 2, or share container if serial).
   - Test 1: Verify API is working initially.
   - Test 2: Modify a protected file inside the running container:
     ```typescript
     execSync('docker exec --user root api-security-test sh -c "echo tampered >> /app/apps/kerala_delivery/api/main.py"');
     ```
     Then send 10+ requests (REVALIDATION_INTERVAL=10) to trigger re-validation. The container should detect integrity violation and exit (SystemExit). Verify either: (a) container has exited (`docker inspect --format='{{.State.Status}}'`), or (b) subsequent requests fail/timeout.
   - afterAll: Stop and remove container (may already be stopped).

   **IMPORTANT NOTE on Scenarios 2 and 3:** These tests require a production-mode container with a POPULATED `_INTEGRITY_MANIFEST` (non-empty dict). In development mode, the manifest is empty and integrity checks are skipped. The test container must be built from the distribution pipeline (`build-dist.sh`) or use a custom Dockerfile stage that populates the manifest. If this is not feasible in the test environment (because `build-dist.sh` requires the full Cython pipeline), then:
   - For Scenario 2 (re-validation): Test that the counter mechanism works by verifying the API stays healthy after N requests. The actual integrity check is skipped in dev mode, but the counter/re-validation logic still executes.
   - For Scenario 3 (integrity tamper): This test may need to be marked as requiring a production build. Add a `test.skip()` condition that checks if `_INTEGRITY_MANIFEST` is populated (query /health or a custom endpoint). If skipped, add a clear message: "Requires production build (build-dist.sh) -- skipped in dev mode."

   The executor should investigate whether the test Docker build produces a container with a populated manifest. If not, document the limitation and test what is testable.

   **Scenario 4: License Renewal via File Drop**
   - beforeAll: Generate an EXPIRED license key (use Python inline to call `encode_license_key()` with `expires_at` 10 days in the past). Also generate a valid RENEWAL key using `generate_license.py --customer "renewal-test" --this-machine --months 1 --verify` and the `--renew` flag if available, otherwise generate a fresh valid key. Write both to temp files.
   - Test 1: Start container with expired license key. Verify GET /api/routes returns 503 (expired beyond grace period).
   - Test 2: Copy renewal.key into container:
     ```typescript
     execSync('docker cp /tmp/renewal-test.key container:/app/renewal.key');
     ```
     Restart container:
     ```typescript
     execSync('docker compose -f docker-compose.yml -f docker-compose.license-test.yml restart api-security-test');
     ```
     Wait for container readiness. Verify GET /health shows `license.status = "valid"` and GET /api/routes returns 200.
   - afterAll: Stop and remove container. Clean up temp key files.
