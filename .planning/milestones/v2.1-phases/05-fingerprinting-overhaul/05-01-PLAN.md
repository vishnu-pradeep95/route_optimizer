<task type="checkpoint:decision" gate="blocking">
  <name>Task 1: Decide MAC address inclusion in fingerprint formula</name>
  <decision>Whether to keep MAC address as a fingerprint signal</decision>
  <context>
The user locked the decision: "Keep existing MAC address signal." However, research confirmed that WSL2 generates a new random MAC address on every reboot (microsoft/WSL#5352). This means:

- INCLUDING MAC: Fingerprint changes on every WSL reboot. Success criterion #1 ("same fingerprint before and after WSL restart") CANNOT be met. Would require license re-generation after every reboot.
- EXCLUDING MAC: Fingerprint is stable across WSL reboots. machine-id + CPU model is sufficient for the threat model (casual copying deterrent). Meets all success criteria.

Research verified: host MAC = 00155de6650d, container MAC = ea77d7280813 (different values). MAC does not help with host-container consistency either.
  </context>
  <options>
    <option id="drop-mac">
      <name>Drop MAC from fingerprint (recommended)</name>
      <pros>Fingerprint stable across WSL reboots. Meets success criterion #1. machine-id + CPU model sufficient for threat model.</pros>
      <cons>Deviates from user's explicit "keep existing MAC address signal" decision.</cons>
    </option>
    <option id="keep-mac">
      <name>Keep MAC in fingerprint (per original decision)</name>
      <pros>Honors user's original decision exactly.</pros>
      <cons>Fingerprint changes on WSL reboot. Success criterion #1 fails. License re-generation needed after every reboot.</cons>
    </option>
  </options>
  <resume-signal>Select: drop-mac or keep-mac</resume-signal>
</task>
