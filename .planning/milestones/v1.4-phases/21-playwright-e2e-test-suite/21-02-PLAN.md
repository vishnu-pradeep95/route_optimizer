```
  });

  // Sequential story tests follow...

```

**Test 1: Upload screen renders correctly**
- Navigate to /driver/
- Clear localStorage: `page.evaluate(() => localStorage.clear())`, then reload
- Assert #upload-section is visible
- Assert "Today's Deliveries" heading visible
- Assert upload button ("Upload Delivery List") visible
- Assert upload icon container visible
- Check browser console for errors (none expected)

**Test 2: CSV upload shows vehicle selector**
- Use `page.waitForEvent('filechooser')` pattern to upload test CSV via the UI button
- Wait for #vehicle-selector to become visible (timeout: 30_000 -- geocoding is slow)
- Assert "Select Your Vehicle" text visible
- Assert at least one vehicle button (.vehicle-btn) exists
- Store vehicle button text content for later assertions

**Test 3: Vehicle selection shows route view**
- Click the first .vehicle-btn
- Wait for #route-view to become visible
- Assert #stop-list has at least one stop card
- Assert #progress-bar is visible
- Assert #header-stats contains text like "0 of N delivered" (or similar)
- Assert navigate button exists (button.btn-navigate)
- Assert Done button exists (button.btn-done)
- Assert Fail button exists (button.btn-fail)
- Assert Call Office FAB (#call-office-fab) is visible
- Store the first stop's order_id from the DOM (data attribute or text)

**Test 4: Mark stop as delivered (UI + API dual verification)**
- Click the first "Done" button (button.btn-done)
- Wait for toast to appear (success toast with "Delivered" text)
- Wait for the DOM to update (toast animation ~1500ms, then re-render)
- Assert the stop is now marked as delivered in the UI (check for visual indicator -- green checkmark, "Delivered" status, or the stop moving to completed section)
- **API verification**: Use `request.get()` to fetch `/api/routes/{vehicleId}` and verify the stop's status changed to "delivered" in the response JSON. This is the dual-check per user decision.
- Assert progress bar updated (e.g., one segment is now green)

**Test 5: Mark stop as failed via dialog modal**
- Click a "Fail" button (button.btn-fail) on the next pending stop
- Assert dialog#fail-dialog is visible (should be a native `<dialog>` element, NOT a browser confirm)
- Assert the dialog has dark backdrop styling
- Assert select#fail-reason dropdown is visible with options
- Assert "Yes, Failed" button (#fail-confirm) is visible
- Assert "Cancel" button (#fail-cancel) is visible
- Click "Cancel" -- assert dialog closes, no state change
- Click "Fail" again on the same stop
- Select a reason from the dropdown (e.g., first non-empty option)
- Click "Yes, Failed" (#fail-confirm)
- Wait for red/error toast to appear
- Assert the stop is now marked as failed in the UI
- **API verification**: Fetch route via API, verify stop status is "failed"

**Test 6: All-done banner (use a minimal vehicle)**
This test needs all stops on a vehicle to be complete. Strategy:
- If there are remaining pending stops on the current vehicle, mark them all as "Done" by clicking btn-done repeatedly (waiting for each toast + re-render)
- Once all stops are marked (mix of delivered/failed from previous tests), assert:
  - #all-done-banner appears with green "Route complete!" text
  - Progress bar is fully filled
  - Banner dismiss button (.all-done-close) exists
  - Click dismiss button -- banner disappears

**Test 7: Navigation flow (reset and re-navigate)**
- Assert the reset button exists (the "reset" or back arrow button that returns to vehicle selector)
- Click reset -- assert #vehicle-selector becomes visible again
- Assert route view is hidden

Important implementation notes:
- ALWAYS clear localStorage before first navigation to prevent stale cached routes (Pitfall 2 from RESEARCH.md)
- Use web-first assertions (expect(locator).toBeVisible()) -- NEVER use page.waitForTimeout() (anti-pattern from RESEARCH.md)
- After clicking Done/Fail, wait for the NEXT expected state rather than hard-coding waits for toast timing
- The PWA reads API key from localStorage('lpg_api_key') -- set this in beforeAll: `page.evaluate((key) => localStorage.setItem('lpg_api_key', key), process.env.API_KEY)`
- Mobile viewport (393x851) is set automatically by the driver-pwa project in playwright.config.ts
