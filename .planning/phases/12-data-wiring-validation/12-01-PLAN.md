**In `apps/kerala_delivery/api/main.py`, in the `update_stop_status` function (after line 1318 `await session.commit()`):**

Add a block that calls `repo.save_geocode_cache` when ALL three conditions are met:
1. `body.status == "delivered"`
2. `delivery_loc is not None` (GPS coordinates present)
3. `target_stop.order.address_raw` is not None (address text exists)

```python
# --- API-07: Save driver-verified location to geocode cache ---
# Only on successful delivery WITH GPS — builds verified Kerala address DB.
# Do NOT fire on "failed" (driver may not be at exact address).
# Do NOT fire without GPS (no location data to save).
if body.status == "delivered" and delivery_loc is not None and target_stop.order.address_raw:
    try:
        await repo.save_geocode_cache(
            session=session,
            address_raw=target_stop.order.address_raw,
            location=delivery_loc,
            source="driver_verified",
            confidence=0.95,
        )
        await session.commit()
    except Exception:
        logger.warning(
            "Failed to save driver-verified location for order %s (non-fatal)",
            order_id,
            exc_info=True,
        )
```

Why call `repo.save_geocode_cache` directly instead of `CachedGeocoder.save_driver_verified`?
- `CachedGeocoder.__init__` requires an `upstream: Geocoder` parameter (for cache-miss lookups)
- The status endpoint has no geocoder instance and doesn't need one
- `save_driver_verified` is just a thin wrapper around `repo.save_geocode_cache` with `source="driver_verified"` and `confidence=0.95`
- Calling the repo function directly with identical parameters achieves the same result without instantiation overhead
- This is a Claude's Discretion item per CONTEXT.md

The `try/except` ensures a geocache save failure never breaks the primary status update flow. The status update + commit already succeeded before this block runs.
