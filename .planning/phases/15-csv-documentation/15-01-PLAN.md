Create `CSV_FORMAT.md` at the project root. The document is written for a NON-TECHNICAL office employee who uploads CSV/Excel files daily. No Python jargon, no developer terminology.

**CRITICAL: Every fact must come from source code, not from memory.** Read each source file listed in context and extract exact values.

**Document structure (task-oriented, NOT code-oriented):**

## 1. Quick Start (2-3 sentences)
- State: "Upload your CDCMS export or a CSV/Excel file. The system accepts .csv, .xlsx, and .xls files up to 10 MB."
- State: "The system automatically detects whether it's a CDCMS export or a standard CSV -- just upload and it figures out the rest."
- Do NOT imply the employee must choose between formats or use different upload buttons.

## 2. CDCMS Export Format (CSV-02)
- Extract the full 19-column header from `cdcms_preprocessor.py` (SAMPLE_CDCMS_HEADER or test file).
- Create a table showing all 19 columns with three categories: "Required by system" (OrderNo, ConsumerAddress), "Used by system" (OrderQuantity, AreaName, DeliveryMan, OrderStatus), "Ignored" (the other 13).
- Document the status filter: only "Allocated-Printed" rows are imported. State this clearly: "Orders with any other status (like 'Delivered' or 'Cancelled') are automatically skipped."
- Document that CDCMS exports are tab-separated and should be uploaded AS-IS without opening in Excel first.

## 3. Standard CSV Format (CSV-03)
- Extract column definitions from `ColumnMapping` class in `csv_importer.py`.
- Create a table with columns: Column Name, Required?, Default If Missing, Notes.
- Emphasize: `address` is the ONLY required column. Everything else is optional.
- List alternative names accepted for address: `delivery_address`, `addr`, `customer_address`.
- Document `cylinder_type` recognized values from `config.py` CYLINDER_WEIGHTS.
- Document coordinate bounds for latitude (6.0-37.0) and longitude (68.0-97.5) from config.py INDIA_COORDINATE_BOUNDS.
- Note: column names are case-insensitive and spaces become underscores automatically.

## 4. Example Rows (CSV-06)
- Extract 2-3 rows from `data/sample_orders.csv` for standard CSV examples.
- Extract 1-2 rows from `data/sample_cdcms_export.csv` for CDCMS examples.
- Show a MINIMAL example (just address column) and a FULL example (all columns).
- Use fenced code blocks with `csv` language hint so the employee can copy-paste.
- Do NOT invent data -- only use rows from the actual sample files.

## 5. What Can Go Wrong (CSV-04)
- Organize by when errors occur: "Before Processing" (file-level) and "During Processing" (row-level).
- Extract every error message from `main.py` (upload endpoint error paths), `csv_importer.py` (_validate_columns, _row_to_order_with_warnings), and `cdcms_preprocessor.py` (_validate_cdcms_columns).
- For each error: What You See (the actual error message text), Why It Happened (plain English), How to Fix It (actionable step).
- Include geocoding errors from GEOCODING_REASON_MAP in main.py.
- Use a simple table or definition-list format, NOT developer error codes.

## 6. Address Cleaning (CSV-05)
- State clearly: "The system automatically cleans CDCMS addresses before looking them up on Google Maps. Standard CSV addresses are NOT cleaned."
- Extract the 10-step pipeline from `clean_cdcms_address()` in `cdcms_preprocessor.py`.
- Show 3-5 before/after examples from `test_cdcms_preprocessor.py` test assertions.
- Document the area suffix: ", Vatakara, Kozhikode, Kerala" is appended automatically.
- Frame as informational ("This is what the system does") not instructional ("You must do this").

## 7. Accepted File Formats (CSV-01)
- This can be a short section near the top or folded into Quick Start.
- List: .csv (comma or tab separated), .xlsx (Excel), .xls (older Excel).
- Maximum file size: 10 MB (from MAX_UPLOAD_SIZE_BYTES in main.py).
- Row capacity: up to 1000 orders per upload.

**Style rules:**
- Use "the system" not "the application" or "the API"
- Use "you" to address the employee
- Use "upload" not "POST" or "send"
- No Python terms: no "ValueError", "DataFrame", "dict", "set notation"
- Keep sentences short. Use numbered steps where appropriate.
- Use tables for structured data (columns, errors). Use prose for explanations.
