    **Part D: Update navigateTo function**

    Update the navigateTo function to accept an optional address parameter for fallback:

    ```javascript
    function navigateTo(lat, lon, address) {
        let url;
        if (lat && lon && lat !== 0 && lon !== 0) {
            // Coordinates for precise routing (primary)
            url = `https://www.google.com/maps/dir/?api=1&destination=${lat},${lon}&travelmode=driving`;
        } else if (address) {
            // Fallback: use address text if coordinates unavailable
            url = `https://www.google.com/maps/dir/?api=1&destination=${encodeURIComponent(address)}&travelmode=driving`;
        } else {
            // No navigation possible
            return;
        }
        window.open(url, '_blank');
    }
    ```

    Update ALL call sites of navigateTo() to pass the address as third argument:
    ```javascript
    navigateTo(stop.latitude, stop.longitude, stop.address)
    ```

    Per user decision: coordinates are the primary destination (precise routing). Address text is only used as fallback when coordinates are missing/zero. Google Maps URLs API does not support a separate label parameter for directions mode, so we use coords for precision.

    **Part E: Null handling for pre-v2.2 routes**

    When `stop.address_raw` is null/undefined (pre-v2.2 routes that don't have address_original stored), the raw address div is simply not rendered. The conditional `stop.address_raw ? ... : ''` handles this. No error states, no "N/A" text -- just hide the secondary line.

    When `stop.address` is null/undefined (should not happen after migration, but defensive), show "Address pending" as fallback text.
