<task type="auto">
  <name>Task 1: Create scripts/stop.sh with graceful stop and --gc mode</name>
  <files>scripts/stop.sh</files>
  <action>
Create `scripts/stop.sh` following the exact boilerplate pattern from start.sh (colors, helpers, SCRIPT_DIR/PROJECT_ROOT resolution, set -euo pipefail).

**Script header block:** Match the style of start.sh/build-dist.sh with description, usage, and duration comments.

**Default mode (no flags):**
1. Get list of running compose services: `docker compose ps --format '{{.Name}}'`
2. If empty, print `info "Services already stopped"` and `exit 0`
3. Store the count and names of running containers
4. Run `docker compose stop`
5. Print each stopped container with checkmark: `success "$name stopped"` (iterate over the previously-captured list)
6. Print total: `success "Stopped $COUNT service(s)"`

**GC mode (`--gc` flag):**
Parse `$1` for `--gc`. When set, execute AFTER the normal stop:

Step 1 -- Orphan container cleanup:
- Run `docker compose down --remove-orphans` (this removes orphan containers and networks, but NOT named volumes since `-v` is omitted)
- Note: this also removes stopped containers from the compose project. That is acceptable for GC mode since we already stopped them.

Step 2 -- Dangling image prune:
- Run `docker image prune -f` and capture output
- Parse "Total reclaimed space:" from output
- Report: `success "Removed N dangling image(s) (SIZE)"` or `info "No dangling images to remove"` if none

Step 3 -- Container log truncation:
- IMPORTANT: Collect log paths BEFORE `docker compose down` removes containers. Restructure the GC flow:
  - First: capture log paths from all containers (running or stopped) using `docker ps -aq` and `docker inspect --format='{{.LogPath}}'`
  - Then: run `docker compose stop` (default mode)
  - Then: run `docker compose down --remove-orphans`
  - Then: truncate the previously-captured log paths
- For each log path, get size with `stat -c%s`, then `sudo truncate -s 0`
- Sum freed bytes. Convert to human-readable with `numfmt --to=iec`
- Report: `success "Truncated container logs (SIZE freed)"` or `info "Container logs already empty"`
- Handle case where sudo is not available: `warn "Log truncation requires sudo -- skipping"`

**Flow restructuring for --gc:** The overall flow when --gc is active:
1. header "Kerala LPG Delivery Route Optimizer"
2. Capture running container names (for stop reporting)
3. Capture ALL container log paths (for later truncation)
4. `docker compose stop` (halt services, report each)
5. header "Garbage collection"
6. `docker compose down --remove-orphans` (clean orphans/networks)
7. `docker image prune -f` (dangling images)
8. Truncate log files from step 3
9. Summary line

**Make executable:** `chmod +x scripts/stop.sh`

**Anti-patterns to AVOID per user decision:**
- Do NOT use `docker compose down -v` anywhere -- this destroys named volumes (pgdata, dashboard_assets)
- Do NOT use `docker image prune -a` -- only dangling (no `-a` flag)
- Do NOT add a confirmation prompt for --gc -- user explicitly opted in
- Do NOT hardcode container names -- use `docker compose ps` and `docker ps -aq` for discovery
