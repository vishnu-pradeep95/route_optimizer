```bash
poll_health() {
    local url="$1" max_wait="$2" elapsed=0
    local spinners=("⠋" "⠙" "⠹" "⠸")
    while [ "$elapsed" -lt "$max_wait" ]; do
        if curl -sf "$url" >/dev/null 2>&1; then return 0; fi
        local idx=$(( (elapsed / 3) % 4 ))
        printf "\r  %s Waiting... (%ds/%ds)  " "${spinners[$idx]}" "$elapsed" "$max_wait"
        sleep 3; elapsed=$((elapsed + 3))
    done
    echo ""; return 1
}
```

  <action>
Create `scripts/verify-dist.sh` following the exact boilerplate pattern from start.sh/build-dist.sh (colors, helpers, SCRIPT_DIR/PROJECT_ROOT resolution, set -euo pipefail).

**Script header block:** Match the style of build-dist.sh with description, usage, output comments.

**Usage:** `./scripts/verify-dist.sh dist/kerala-delivery-v1.4.tar.gz`
Accept tarball path as `$1`. Validate it exists and is a .tar.gz file.

**Constants:**
```bash
VERIFY_PORT=8002
VERIFY_PROJECT="verify-dist"
MAX_WAIT=120  # 2 minutes (longer than start.sh's 60s because dashboard must build)
POLL_INTERVAL=3
```

**Step 1/5: Extract tarball to temp directory**
- `VERIFY_DIR=$(mktemp -d)`
- Set up trap: `trap 'cleanup' EXIT`
- `tar xzf "$TARBALL" -C "$VERIFY_DIR"`
- Verify `$VERIFY_DIR/kerala-delivery/docker-compose.yml` exists
- `EXTRACT_DIR="$VERIFY_DIR/kerala-delivery"`
- `cd "$EXTRACT_DIR"`

**cleanup() function:**
```bash
cleanup() {
    info "Cleaning up verification environment..."
    if [ -d "$EXTRACT_DIR" ]; then
        cd "$EXTRACT_DIR" 2>/dev/null || true
        docker compose --project-name "$VERIFY_PROJECT" down --volumes --remove-orphans 2>/dev/null || true
    fi
    rm -rf "$VERIFY_DIR"
    success "Cleanup complete"
}
```

**Step 2/5: Generate .env**
- Copy .env.example to .env
- Generate random DB password and API key using `tr -dc 'A-Za-z0-9' < /dev/urandom | head -c 32`
- Use sed to replace values in .env:
  - `POSTGRES_PASSWORD=<random>`
  - `API_KEY=<random>`
  - `ENVIRONMENT=development`
  - `GOOGLE_MAPS_API_KEY=` (leave empty -- not needed for verification)
- Print: `success ".env generated with dummy credentials"`

**Step 3/5: Generate compose port override**
Create `docker-compose.verify.yml` in `$EXTRACT_DIR`:
```yaml
services:
  db:
    ports:
      - "5433:5432"
  osrm:
    ports:
      - "5001:5000"
  vroom:
    ports:
      - "3001:3000"
  api:
    ports:
      - "8002:8000"
    environment:
      - OSRM_URL=http://osrm:5000
      - VROOM_URL=http://vroom:3000
```
Print: `success "Port overrides generated (API on :$VERIFY_PORT)"`

**Step 4/5: Build and start services**
- Create data directories: `mkdir -p data/osrm data/geocode_cache`

- OSRM handling (Claude's discretion per CONTEXT.md): Skip OSRM/VROOM entirely for speed. The /health endpoint, /driver/, and /dashboard/ do not require OSRM. Start only the essential services:
  ```bash
  docker compose \
      --project-name "$VERIFY_PROJECT" \
      -f docker-compose.yml \
      -f docker-compose.verify.yml \
      up -d --build db db-init dashboard-build api
  ```
  Note: The `api` service has `depends_on` for osrm/vroom with `condition: service_started`. Since we are NOT starting osrm/vroom, we need to handle this. The compose file requires them. Two approaches:

  **Approach A (recommended):** Add to docker-compose.verify.yml an override that removes the osrm/vroom dependency from api. Docker Compose override files can override `depends_on`:
  ```yaml
  api:
    depends_on:
      db-init:
        condition: service_completed_successfully
      dashboard-build:
        condition: service_completed_successfully
  ```
  This replaces the entire `depends_on` block for api, removing the osrm/vroom requirements.

  **Approach B (fallback):** Start ALL services but accept that osrm-init may hang. Then only verify the 3 endpoints. Use `--timeout 120` on compose up.

  Go with Approach A. The override file becomes:
  ```yaml
  services:
    db:
      ports:
        - "5433:5432"
    api:
      ports:
        - "8002:8000"
      depends_on:
        db-init:
          condition: service_completed_successfully
        dashboard-build:
          condition: service_completed_successfully
  ```
  (Remove osrm and vroom service overrides entirely since we are not starting them.)

- Poll health using spinner pattern from start.sh: `poll_health "http://localhost:$VERIFY_PORT/health" "$MAX_WAIT"`
- On failure: print diagnostic with `docker compose --project-name "$VERIFY_PROJECT" logs --tail=30` and exit 1

**Step 5/5: Verify endpoints**
Track PASS/FAIL counts:
- `GET /health` -- curl returns 200
- `GET /driver/` -- curl response contains "html" (case-insensitive)
- `GET /dashboard/` -- curl response contains "html" (case-insensitive)

Print results with success/error per check. At the end:
- If all 3 pass: print green banner (matching start.sh style) -- "Distribution verified! All 3 checks passed."
- If any fail: print which failed, show diagnostic logs, exit 1

**Cleanup:** Handled by trap. On normal exit, cleanup tears down the verification stack and removes temp dir.

**Make executable:** `chmod +x scripts/verify-dist.sh`
